How do I route between screens?

With components and lw-if — no router. leanweb.urlHash mirrors location.hash, so screens are just conditions:


         <demo-login lw-if="leanweb.urlHash === '#/login'"></demo-login>
         <demo-signup lw-if="leanweb.urlHash === '#/signup'"></demo-signup>
         <demo-dashboard lw-if="leanweb.urlHash === '#/dashboard'"></demo-dashboard>

Define urlHashChanged() on the routing component: the runtime calls it on every hash change and re-renders the component, which re-evaluates the conditions. Parked screens keep their state for the way back.


         customElements.define('demo-root',
            class extends LWElement {
               urlHashChanged() {
                  document.title = 'demo ' + leanweb.urlHashPath;
               }
            }
         );

How do I update a component from outside?

Components are real elements, so the direct route always works:


         document.querySelector('demo-login').update();

By tag name, without touching the DOM — or app-wide with no arguments. Only changed parts are written:


         leanweb.updateComponents('demo-login', 'demo-cart');
         leanweb.updateComponents();

How do environments work?

lw serve and lw dist ship src/env.js as-is; lw serve prod and lw dist prod swap in src/env/prod.js. Add files to src/env/ for as many environments as you need.


         export default {
            apiUrl: 'https://api.example.com'
         };
      

         import env from './../../env.js';

         const users = await fetch(env.apiUrl + '/users');

What can the dev server do?

Options are environment variables in front of the command:

$ port=9999 lw serve
$ host=0.0.0.0 lw serve # all IPv4 interfaces
$ host=:: lw serve # IPv6 too
$ noopen=1 lw serve # skip opening the browser
$ https=./https.js lw serve
$ verbose=1 lw dist # bundle analysis

         const fs = require('fs');

         module.exports = {
            cert: fs.readFileSync(__dirname + '/localhost.pem'),
            key: fs.readFileSync(__dirname + '/localhost.key.pem'),
            passphrase: '12345'
         };

Anything after the build?

If an executable post-dist file sits in the project root, lw dist runs it after the build lands — handy for shipping dist/ wherever it goes next. This site's own post-dist copies robots.txt and sitemap.xml into place.


         #!/bin/sh
         cp -r dist/* ../server/public/

Need Shadow DOM after all?

Start the project with lw init --shadow-dom, or flip shadowDom to true in leanweb.json. Same directives, same components — CSS moves into adopted stylesheets, global-styles.css reaches into every shadow root, and the native <slot> takes over projection from lw-slot.


         {
            "name": "demo",
            "shadowDom": true
         }
