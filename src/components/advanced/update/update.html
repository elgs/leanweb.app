Who renders when: updateComponents#

A component re-renders when it initializes, a local lw-on: event fires, a local lw-model changes, a local async method settles, or its parent re-renders — updates propagate down. In short: data flows down, signals go up, and across is up-then-down. For everything with no DOM position — timers, sockets, module code — you say when: this.update() for one component, leanweb.updateComponents(...tagNames) for others, no arguments for the whole app.


            <demo-controls></demo-controls>
            <demo-red></demo-red>
            <demo-red></demo-red>
            <demo-blue></demo-blue>
         

            customElements.define('demo-controls',
               class extends LWElement {
                  all() { leanweb.updateComponents(); }
                  red() { leanweb.updateComponents('demo-red'); }
                  blue() { leanweb.updateComponents('demo-blue'); }
               }
            );
         

            customElements.define('demo-red',
               class extends LWElement {
                  constructor() {
                     super(ast);
                     setInterval(() => {
                        this.time = new Date().toLocaleTimeString();
                     }, 1000);
                  }
               }
            );
         

            <p>red <b lw>time ?? '-'</b></p>

These clocks tick every second but never call update() — their DOM refreshes only when asked. The buttons live in their own component on purpose: a click re-renders that component alone, and the clocks are its siblings, not its children — so they refresh only through updateComponents. demo-blue is demo-red with a different tag.
