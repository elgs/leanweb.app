lw-slot — content projection#

A component with an <lw-slot> in its template renders whatever its client — the component using it — writes between its tags. The component supplies the chrome; the client supplies the content. And the content keeps belonging to the client: its expressions, its handlers, its updates. That's what makes generic wrappers — cards, dialogs, panels — reusable.


            <div class='chrome' lw>title</div>
            <div class='content'>
               <lw-slot></lw-slot>
            </div>
         

            customElements.define('demo-card',
               class extends LWElement {
                  title = 'demo-card chrome';
               }
            );
         

            <demo-card>
               <p>Clicked <b lw>clicks</b> times.</p>
               <button lw-on:click='clicks++'>Click me</button>
            </demo-card>

            <demo-card>
               <input lw-model='note' placeholder='type a note'>
               <p lw>note</p>
            </demo-card>
         

            customElements.define('demo-app',
               class extends LWElement {
                  clicks = 0;
                  note = '';
               }
            );

Named slots & fallback content

A template can offer more than one slot: give each a name, and leave at most one unnamed — the default. A client child with slot='name' lands in the matching <lw-slot name='name'>; everything else lands in the default slot. And whatever the component writes inside a slot is fallback content — its own markup, rendered by itself — kept only when the client sends nothing for that slot.


            <div class='chrome'>
               <span>demo-pane</span>
               <lw-slot name='actions'></lw-slot>
            </div>
            <div class='content'>
               <lw-slot></lw-slot>
            </div>
            <div class='foot'>
               <lw-slot name='footer'>
                  <small>demo-pane default footer</small>
               </lw-slot>
            </div>
         

            customElements.define('demo-pane',
               class extends LWElement {
                  // no state, no handlers — just placement
               }
            );
         

            <demo-pane>
               <button slot='actions' lw-on:click='star()'>Star</button>
               <p>Starred <b lw>stars</b> times. No footer sent —
                  the pane keeps its own.</p>
            </demo-pane>

            <demo-pane>
               <button slot='actions' lw-on:click='reset()'>Reset</button>
               <p>This one sends a footer of its own.</p>
               <small slot='footer' lw>'stars: ' + stars</small>
            </demo-pane>
         

            customElements.define('demo-app',
               class extends LWElement {
                  stars = 0;

                  star() {
                     this.stars++;
                  }

                  reset() {
                     this.stars = 0;
                  }
               }
            );

One slot='actions' attribute put each button in the chrome — the pane never declared either. The left footer is the pane's fallback; the right one is client content that replaced it, and it keeps rendering client state. Names pick the landing spot; ownership never moves.

Talking across the slot

The tag is the component's API. Data flows in through lw-input: — the client hands the panel an object once, and the panel reads it on every render. Signals flow out as DOM events the client catches with lw-on: on the tag. Nothing here is new to projection — the slot moves markup, not data.


            <div class='chrome'>
               <span lw>header.title</span>
               <button lw-on:click='refresh()'>Refresh</button>
            </div>
            <div class='content'>
               <lw-slot></lw-slot>
            </div>
         

            customElements.define('demo-panel',
               class extends LWElement {
                  header = { title: 'demo-panel' };

                  refresh() {
                     this.dispatchEvent(new CustomEvent('refresh'));
                  }
               }
            );
         

            <demo-panel lw-input:header='header' lw-on:refresh='reload()'>
               <p lw>'Loaded at ' + loadedAt</p>
               <button lw-on:click='renamePanel()'>Rename the panel</button>
            </demo-panel>
         

            customElements.define('demo-app',
               class extends LWElement {
                  header = { title: 'Results' };
                  renames = 0;
                  loadedAt = new Date().toLocaleTimeString();

                  reload() {
                     this.loadedAt = new Date().toLocaleTimeString();
                  }

                  renamePanel() {
                     this.header.title = 'Results v' + (++this.renames);
                  }
               }
            );

'Loaded at ' + loadedAt

Refresh sits on the panel's chrome, but what it reloads is the client's content. Rename sits in the content, but what it changes is the panel's chrome. Rules: at most one lw-slot per name — one unnamed default — and never inside an lw-for; both are build errors. With shadowDom: true the native <slot> does this job, names included; lw-slot brings the same model to the default light DOM. Every live frame on this page is a component with an lw-slot.
